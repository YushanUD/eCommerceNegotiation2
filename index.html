Welcome - You are building a real web app

You are about to build your own E-Commerce Negotiation Platform where a shopper and a computer agent can exchange offers. You do not need to write code from scratch. You will guide AI clearly, and it will generate the code for you.

What you are building

Your app lets an Admin configure a negotiation topic (like buying a laptop), define features and option values, and set preference weights. Then Users negotiate with an AI-style seller agent by sending and receiving offers. Each offer stores choices, price, and utility score so negotiation outcomes can be reviewed later.

Simple picture of the final experience

  • Admin signs in and creates negotiation templates and agent preferences
  • User signs in, sets personal preferences, and starts a negotiation session
  • User and agent exchange offers until user accepts or ends the negotiation

Understanding dynamic web apps (deep explanation)

Website vs web app

A normal website is like a printed brochure: mostly read-only information. A web app is like an interactive service desk: people log in, submit information, and get personalized results. Your project is a web app because users have accounts, data is saved, and each negotiation depends on user input.

Client-server model using a restaurant analogy

Imagine a restaurant. The dining room is what customers touch and see. The kitchen stores ingredients and prepares meals. In your app, the frontend (client) is the dining room in the browser. The backend (server) is the kitchen where data is stored and processed. The API is the waiter carrying requests and responses.

Why both frontend and backend matter

  • Persistence: data must survive when the browser closes
  • Multiple users: different users need one shared source of truth
  • Security: permission checks should happen outside the browser

What this means for your app

  • Next.js + React: builds interactive pages users click and type into
  • InstantDB: provides backend data + authentication without building a server from zero
  • Vercel: puts your finished app online so others can use it

Welcome to Your App-Building Journey

You are about to build a real web app: an e-commerce negotiation platform where students negotiate with a computer agent, and you (as admin) configure products and agent preferences. By the end, your app can be live on the internet and used by real people.

What You Are Building

Your app helps online shoppers negotiate product offers instead of accepting one fixed price. Students create their own preference profile (what matters most to them), then exchange offers with a rules-based computer agent. You can configure product options and the agent's preference weights as the admin.

Simple Finished-App Picture

  • Login with email and magic code
  • Student selects product preferences and option weights
  • Negotiation screen shows offer history and counter-offers
  • Admin screen configures product features and agent preference logic

Understanding Dynamic Web Apps

Website vs Web App

A website is usually like a digital brochure: you read content. A web app is interactive: people sign in, save data, and perform actions. Your project is a web app because each user has an account and their own saved preferences and negotiations.

The Client-Server Model (Restaurant Analogy)

Imagine a restaurant. The dining room is where customers interact. The kitchen is where food is prepared and stored. Customers do not enter the kitchen directly.
  • Frontend (client): The dining room. Buttons, forms, and screens users see.
  • Backend (server): The kitchen. Stores data and enforces rules.
  • API: The waiter carrying requests and responses between both sides.

Why Both Frontend and Backend Matter

  • Persistence: Data survives after browser close.
  • Multiple users: Everyone can access shared product definitions, but keep private profiles.
  • Security: Permission checks happen in the backend where users cannot tamper.

What This Means for Your Project

  • Frontend with Next.js + React for interactive screens.
  • Backend and auth with InstantDB using magic code login.
  • No custom backend server needed for v1; InstantDB handles storage and auth.
