Your Personalized Build Prompt

Build me a Next.js web application. Here are the complete specifications:

## Tech Stack (DO NOT DEVIATE)
- Next.js 14+ with App Router (not Pages Router)
- TypeScript
- Tailwind CSS for styling
- InstantDB for database and authentication (use @instantdb/react)

## InstantDB Configuration
My InstantDB App ID is: YOUR_APP_ID_HERE

Initialize InstantDB in a file at `src/lib/db.ts` like this:
```typescript
import { init } from '@instantdb/react';
import schema from '../../instant.schema';

const APP_ID = 'YOUR_APP_ID_HERE';

export const db = init({
  appId: APP_ID,
  schema
});
```

## App Description
Build an e-commerce negotiation platform where students negotiate product offers with a rules-based computer agent.
The admin user configures products, negotiable features, and the agent's feature preference weights.
Each student user configures their own personal feature and option preferences, then exchanges offers and can accept a final offer.

## Roles and Access Rules
- There are two user roles: `admin` and `student`.
- The admin is me, identified by my login email (I will provide it in env/config as ADMIN_EMAIL).
- Admin can create/edit products, features, options, and agent preference profiles.
- Student users can configure only their own preference profile and negotiation sessions.
- Enforce role-based restrictions in UI and InstantDB permissions.

## Data Model
Create an InstantDB schema in `instant.schema.ts` with these entities:
- `profiles`: role (string: "admin" | "student"), fullName (string), ageRange (string), gender (string), createdAt (date)
- `products`: name (string), description (string), isActive (boolean), createdAt (date)
- `features`: name (string), weightPercent (number), createdAt (date)
- `featureOptions`: label (string), utilityWeight (number), createdAt (date)
- `agentPreferenceProfiles`: name (string), notes (string), createdAt (date)
- `studentPreferenceProfiles`: name (string), createdAt (date)
- `offers`: roundNumber (number), offeredPrice (number), notes (string), status (string: "pending" | "accepted" | "rejected"), createdAt (date)
- `negotiationSessions`: status (string: "active" | "completed"), finalDecision (string: "accepted" | "walked_away" | "ongoing"), createdAt (date)

And these relationships:
- Each product has many features.
- Each feature has many featureOptions.
- Each agentPreferenceProfile belongs to one product and has feature-level weights.
- Each studentPreferenceProfile belongs to one user and one product.
- Each negotiationSession belongs to one student user and one product.
- Each negotiationSession has many offers.
- Each offer belongs to exactly one negotiationSession and one creator user.

## Authentication
Implement user authentication using InstantDB Magic Code flow:
1. User enters email
2. InstantDB sends a 6-digit code
3. User enters code to log in
4. Use `db.useAuth()` to check status
5. Protect all app routes except login

## Pages to Create
- `/` - Home dashboard with role-based quick actions and active negotiations
- `/login` - Email + magic code login page
- `/admin/products` - Admin page to create and manage products
- `/admin/products/[id]/features` - Admin page to manage negotiable features and options
- `/admin/agent-preferences` - Admin page to define computer agent preference weights
- `/profile` - Student demographic/profile setup page
- `/preferences` - Student preference setup page (feature weights sum to 100%)
- `/negotiation/new` - Start a negotiation session for a selected product
- `/negotiation/[id]` - Negotiation room with offer history and counter-offers
- `/my-negotiations` - Student list of current and past negotiation sessions

## Core Features
1. Admin can set products and negotiable feature structures.
2. Admin can configure the computer agent preference profile and acceptance thresholds.
3. Student can set personal preference weights (must total 100%) and option utilities.
4. Student and agent exchange offers by round.
5. Rules-based agent creates counter-offers from weighted scoring (no external APIs).
6. Agent accepts when offer score reaches threshold; student can accept or continue.
7. Every user sees only data they are allowed to view based on role and ownership.

## Rules-Based Negotiation Engine Requirements
- Implement deterministic logic in code, no external AI services.
- Calculate weighted utility score for each offer using:
  - feature weight
  - selected option utility
  - price sensitivity rule
- Return one of: counter-offer, accept, or reject.
- Show a transparent explanation panel in UI: why the agent responded that way.
- Keep logic simple and well-commented so a beginner can follow.

## UI Requirements
- Clean, modern Tailwind design
- Mobile responsive
- Clear role-aware navigation
- Loading states for async operations
- Friendly empty states
- Helpful validation errors (especially for weight totals)

## Getting Started Steps
1. Create app:
   `npx create-next-app@latest . --tailwind --eslint --app --src-dir --import-alias "@/*" --typescript`
2. Install dependencies:
   `npm install @instantdb/react`
3. Create schema, perms, and db initialization files.
4. Implement auth flow and role bootstrap.
5. Build pages and shared components.
6. Add deterministic negotiation engine logic.
7. Ensure app runs with `npm run dev`.

Please create all necessary files with complete, working code, and include clear comments for major sections since I am a beginner.

3) Run Your App Locally

npm install
npm run dev

Open http://localhost:3000 in your browser. Localhost means "run on this computer only." It is your private test environment before deployment.

4) If You See Errors

  1. Copy the exact error text from terminal or browser console.
  2. Paste it into Cursor Composer.
  3. Ask: I got this error. Please fix it and explain why it happened in beginner terms.
